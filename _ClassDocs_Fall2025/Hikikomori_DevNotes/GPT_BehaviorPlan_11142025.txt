2. Adapting your BehaviorSystem to lean more like CS

Here I’ll sketch an updated architecture: what changes or extensions to your existing API, what new components, and how the flow works end-to-end.

2.1 High-level flow

In the house scene, you have multiple “BehaviorAnchors” (3D GameObjects) placed at e.g. desk, kitchen, bathroom. These serve as spatial anchors.

On screen: if a BehaviorAnchor is within the camera’s view frustum, you overlay a fixed-screen-space UI icon (flat UI) tied to that anchor (so world position → screen position). If the anchor is off-screen, no icon.

Player clicks the icon → camera transitions (zooms in) to that anchor (so your existing camera system takes over maybe with an animation). Then you present the BehaviorPanel (UI) listing the available behaviors for that anchor’s “room” context.

Player selects a BehaviorData item: we check unlocks, resource costs, etc. Then:

If BehaviorData.isScene == false → one-shot: brief progress meter, subtract cost/time/spoons, trigger direct thought (Yarn node maybe) and return to idle.

If BehaviorData.isScene == true → interactive dialogue: pause clock/time, start Yarn (dialogueRunner) node(s), wait for interactive choices, behind the scenes you may deduct resources and set branching flags/outcomes, then resume clock and return to idle (or maybe go to next default behavior).

Meanwhile default idle behavior is running (your defaultBehavior) when player is “not doing anything else.” Perhaps during idle we can still accumulate some passive change (time, spoons regen or hunger increase) until a behavior is triggered.

We maintain a “cycle” or “time block” unit (you already have minutes → cost/time). We can optionally incorporate something like “actions per cycle” or “daily reset” if you want to mirror CS more strictly (but you don’t have to copy that exactly).

BehaviorManager modifications

3. Pros, Cons & Implementation challenges
Pros

You get a richer “decision space” while playing that fits your narrative of managing limited spoons/time.

The spatial anchors + camera zoom provide nice flavour and scene immersion (player locates the behavior in world).

The branching scene vs one-shot distinction adds variety: quick pickups vs deeper story moments.

Borrowing CS’s rhythm (limited resources + meaningful actions) gives your game a compelling structure.

Cons / challenges

The CS system is fairly abstract (dice + cycles) and your system is more literal (behaviors in house). You’ll need to ensure you don’t get bogged down by too many small tasks that feel like busy-work.

Balancing: lots of behaviors might tempt players to “grind” easy ones just to clear resources; you’ll need to design spoon/time/hunger/cash cost curves carefully.

Implementation complexity: The anchor → UI icon mapping, camera transitions, and handling off-screen/hide logic all add architectural overhead.

Branching behaviour scenes: writing enough Yarn dialogue, handling outcomes, and ensuring state persists can get tricky.

Visual clarity: making sure the icons don’t clutter, the player understands what’s available, and “why I should click this now” is clear.

Specific technical challenges

Converting world position → screen space and handling click vs off-screen hide: you’ll need to consider UI canvas vs world camera, maybe using Camera.WorldToScreenPoint() and some screen bounds checks.

Making sure the camera zoom doesn’t conflict with your existing camera system (you said you have spatial camera system switching angles). You’ll need a clean way to “lock” the camera during behavior transitions and restore.

Pausing/resuming the clock while dialogues happen: you already handle pause in RunYarnNode, good, but even in one-shot you might want to freeze certain inputs or behaviours.

Behavior queueing: you already have a queue for queued behaviors; you may want to extend to “actions per cycle” logic (e.g., when queue empty and availableActions==0 → force cycle end). Ensuring UI updates accordingly.

Unlock logic: maintaining good feedback to the player that a behavior is locked/unlocked, maybe UI overlay states etc.

4. Suggested Next Step Implementation Plan

Here’s how you might roll this out in your project:

Create BehaviorAnchor prefab: For each room (desk, kitchen, bathroom) add a GameObject with BehaviorAnchor component, assign roomType, assign icon sprite, assign list of BehaviorData for that anchor.

World → screen UI icon system: Create a manager (e.g., BehaviorIconUIManager) that cycles through all BehaviorAnchor instances, does WorldToScreenPoint, checks if point is within screen bounds (and maybe anchor not occluded) and if yes, instantiate/update a UI icon (on a Screen-Space canvas, fixed size) at that position; if off-screen, hide the icon. Clicking the icon triggers BehaviorAnchor.Clicked event.

Camera transition logic: Extend your existing system to take a target Transform (BehaviorAnchor.transform) and animate camera/movement to a fixed zoom position/angle. When done, callback to open UI panel. Also handle returning to default viewpoint when panel closes.

BehaviorPanel adaptation: Link BehaviorAnchor to BehaviorPanel: when BehaviorAnchor is clicked, pass behaviors list to BehaviorPanel.PopulateGrid for the room, open panel. The rest of BehaviorPanel code (select room, etc) you might simplify: since it’s always context of the anchor, you can skip multiple room tabs and just show behaviors for that anchor. Or keep room tabs if you want categories.

BehaviorManager adjustments:

Add isScene flag to BehaviorData; use it to branch logic in StartBehavior.

Add an actionsRemaining field (optional) to count actions per cycle; reduce when behaviors used; integrate into HandleClockTick to reset when cycle ends.

Modify TryStartBehavior or StartBehavior to check resources (spoons, hunger, cash, actions) before proceeding, and give feedback.

Extend RunBehavior logic: for one-shot behaviour you might short-circuit the while loop, or run a faster progress bar coroutine then subtract cost.

Tie into Yarn for branching: For scene behaviours create Yarn nodes that represent start → choices → end; you already have the triggers logic for OnStart, OnMidpoint, OnEnd in BehaviorData. For one-shot behaviours you might still use OnEnd to show a quick thought.

Design & balance: Create a set of behaviours, assign realistic costs/time/spoons, decide which are one-shots vs scenes, map them to anchors. Play-test to ensure the player is facing meaningful trade-offs.

Cycle and reset logic: If you adopt actions per cycle, you’ll need a “day end” trigger (maybe when player returns to bed or when actionsRemaining==0) that resets availableActions, maybe reduces hunger/other stats, triggers any end-of-day events, then refreshes.

UI feedback: Show icons that grey out when locked/in cooldown, show tooltips when hovering/pointer (you already have ToolTipPanel). For scene behaviours you might temporarily disable other anchors until the panel closes.

Persistence: Save unlocked behaviours, resource states, time-of-day, progress. Ensure load/quit handles mid-behavior gracefully.
